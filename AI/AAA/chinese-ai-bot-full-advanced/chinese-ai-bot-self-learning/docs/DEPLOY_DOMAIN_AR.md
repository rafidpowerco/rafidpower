# نشر التطبيق على دومينك (تجربة مقرّبين)

هذا الدليل عملي لجعل الموقع يفتح من عنوانك `https://example.com` مع HTTPS وجلسات آمنة.

## 1) تجهيز الخادم

- خادم لينكس أو VPS مع عنوان عام ثابت (أو اسم نطاق يشير إليه).
- Node 20+ و `pnpm`، أو تشغيل عبر Docker كما في `docker-compose` في المشروع.
- MySQL متاح و`DATABASE_URL` صحيح؛ بعد تحديث المخطط نفّذ الترحيلات:
  - `DATABASE_URL=... pnpm db:migrate`
  - يشمل جدول آراء المستخدمين بعد إضافة `0006_user_feedback.sql`.

## 2) DNS

في لوحة النطاق:

- سجل **A** (أو **AAAA** لـ IPv6): الاسم `@` أو `chat` → يشير إلى IP الخادم.
- انتظر انتشار DNS (غالباً من دقائق إلى ساعات).

## 3) بناء وتشغيل التطبيق

من جذر المشروع:

```bash
pnpm install
pnpm build
```

التشغيل الإنتاجي (مثال):

```bash
NODE_ENV=production PORT=3000 HOST=0.0.0.0 pnpm start
```

أو استخدم **PM2** / **systemd** لإعادة التشغيل التلقائي. خلف عكس وكيل (الخطوة التالية) غالباً يبقى التطبيق على `127.0.0.1:3000`.

## 4) عكس وكيل و HTTPS (Caddy أو nginx)

**مهم:** الواجهة الأمامية و tRPC تُخدم عادة من نفس المنفذ؛ تأكد أن المسار `/api` يصل للتطبيق كما في التطوير.

### Caddy (مبسّط)

مثال موقع يحصل على شهادة Let’s Encrypt تلقائياً:

```text
chat.example.com {
  reverse_proxy 127.0.0.1:3000
}
```

### nginx

- استمع على 443 مع شهادة SSL.
- `proxy_pass http://127.0.0.1:3000;`
- مرّر الرؤوس: `Host`, `X-Real-IP`, `X-Forwarded-For`, `X-Forwarded-Proto`.

على الخادم، عطّل الثقة بعناوين العميل إلا إذا كنت خلف بروكسي موثوق:

```env
TRUST_PROXY=1
```

## 5) متغيرات البيئة الحرجة

| المتغير                                        | الغرض                                                                             |
| ---------------------------------------------- | --------------------------------------------------------------------------------- |
| `PUBLIC_APP_URL`                               | الرابط العلني `https://chat.example.com` — يُستخدم لتوجيه OAuth والروابط الصحيحة. |
| `JWT_SECRET`                                   | توقيع كوكي الجلسة؛ إلزامي في الإنتاج.                                             |
| `DATABASE_URL`                                 | اتصال MySQL.                                                                      |
| `VITE_DEFAULT_TENANT_ID` / `DEFAULT_TENANT_ID` | موحّدان مع بيئة الاختبار إن استخدمت مستأجراً واحداً.                              |
| `VITE_SHOW_BETA_BANNER=1`                      | اختياري لإظهار شريط الإصدار التجريبي.                                             |

في مزوّد OAuth (Manus أو غيره) سجّل **Redirect URI** مطابقاً للواقع، مثلاً:

`https://chat.example.com/api/oauth/callback`

## 6) جمع آراء المستخدمين

- الصفحة: `/feedback` (يتطلب تسجيل الدخول).
- المسؤول (`role=admin`) يرى آخر الملاحظات في **لوحة التحكم** ويمكنه أيضاً استدعاء `feedback.listRecent` عبر tRPC إن لزم التصدير لاحقاً.

## 7) تحقق سريع

- **`GET /health`** أو **`GET /api/health`**: JSON خفيف لمراقبة التشغيل وقاعدة البيانات (`200` أو `503` إن وُجد `DATABASE_URL` والاتصال فاشل).
- للمسؤول: صفحة **`/status`** في الواجهة ومسار tRPC **`system.healthDeep`** (بدون استهلاك LLM تلقائياً).

- فتح `https://your-domain` يعرض الصفحة الرئيسية.
- تسجيل الدخول يعمل دون خطأ redirect.
- إرسال ملاحظة من `/feedback` ينجح (تحقق من جدول `user_feedback` في MySQL إن رغبت).

لمسائل الامتثال والملكية الفكرية راجع `docs/IP_AND_LEGAL_AR.md` و`THIRD_PARTY_NOTICES.md`.

## 8) يبدو أن الـ API «متوقف» بعد رفع الموقع

- **استضافة ثابتة فقط (Netlify / Pages بدون Node):** الواجهة تُبنى لكن **`/api/trpc` لن يعمل** لأن الخادم غير موجود. الحل: تشغيل تطبيق Node هذا (Docker أو VPS) وتوجيه الدومين إليه، أو فصل الواجهة عن الـ API مع ضبط **`VITE_TRPC_URL`** و **`CORS_ORIGINS`** (انظر `.env.example`).
- **خلف nginx/Caddy بدون `TRUST_PROXY=1`:** كل الزوار قد يُحسبون عنوان IP واحداً → **429** سريعاً على `/api/trpc`. عيّن **`TRUST_PROXY=1`** ومرّر **`X-Forwarded-For`** و **`X-Forwarded-Proto`**.
- **المسار `/api` لا يصل للتطبيق:** تأكد من `reverse_proxy` / `proxy_pass` لكل الموقع وليس للملفات الثابتة فقط.
- **Forge / مزوّد خارجي:** إن قُيّد الوصول بعد النشر، استخدم **`LLM_BACKEND=openrouter`** أو **`custom`** مع خادمك (انظر `.env.example`).
- **إعدادات تجريبية محلية دون Git:** أنشئ في جذر المشروع **`local.runtime.json`** (مُستبعد من `.gitignore`)؛ يُحمّل تلقائياً للمفاتيح الفارغة فقط بعد `.env`.

## 9) أسرار GitHub

لا ترفع `.env` أو رموز تليجرام أو كلمات مرور قاعدة البيانات. استخدم **`local.runtime.json`** أو متغيرات الخادم فقط. القالب **`local.runtime.example.json`** يوضح الشكل دون قيم حقيقية.
